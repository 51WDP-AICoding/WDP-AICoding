# WDP 需求明确化子技能

本技能负责将用户模糊的需求转化为结构化的技术规格，便于AI理解并生成可执行代码。

## 前置条件

1. 已获取用户的初始需求描述。
2. 已确认需求属于WDP技术域范围。

## 标准流程

### 1. 需求分析与分类

首先对用户需求进行分析和分类，确定其所属的技术域：

1. **确定核心技术域**
   - 场景初始化与接入问题 → wdp-api-initialization-unified
   - 相机控制与漫游问题 → wdp-api-camera-unified
   - 实体操作问题 → wdp-api-entity-general-behavior
   - 覆盖物问题 → wdp-api-coverings-unified
   - 图层/模型问题 → wdp-api-layer-models
   - 材质问题 → wdp-api-material-settings
   - 点聚合问题 → wdp-api-cluster
   - 功能组件问题 → wdp-api-function-components
   - BIM问题 → wdp-api-bim-unified
   - GIS问题 → gis-api-core-operations
   - 空间理解问题 → wdp-api-spatial-understanding
   - 事件注册问题 → wdp-api-general-event-registration
   - 属性操作问题 → wdp-api-generic-base-attributes

2. **识别跨域需求**
   - 如果需求涉及多个技术域，确定主要域和次要域
   - 按照依赖关系排序（如初始化必须先于其他操作）

### 2. 关键参数提取

从用户需求中提取关键参数：

1. **必要参数**
   - 服务器URL（如果未提供，需要追问）
   - 验证口令（如果未提供，需要追问）
   - 渲染容器ID（如果未提供，可以默认为"player"）
   - 特定实体ID/名称（如果操作特定对象，需要追问）

2. **可选参数**
   - 视觉效果参数（颜色、大小、透明度等）
   - 动画参数（持续时间、速度等）
   - 交互参数（点击行为、悬停效果等）

3. **参数默认值处理**
   - 对于未明确指定的非必要参数，使用合理的默认值
   - 在代码注释中标明哪些参数使用了默认值

### 3. 依赖关系识别

识别需求实现过程中的依赖关系：

1. **技术依赖**
   - SDK依赖（WdpApi、CloudApi、BimApi、GisApi等）
   - 插件依赖（如BIM操作需要先安装BIM插件）

2. **时序依赖**
   - 初始化 → 场景就绪 → 业务操作
   - 对象创建 → 对象配置 → 对象控制

3. **数据依赖**
   - 数据加载 → 数据处理 → 数据展示
   - 用户输入 → 数据验证 → 操作执行

### 4. 执行计划制定

将分析结果转化为结构化的执行计划：

1. **初始化阶段**
   - SDK加载
   - 实例创建
   - 渲染启动
   - 场景就绪监听

2. **准备阶段**
   - 插件安装（如需要）
   - 数据加载
   - 基础环境设置

3. **业务实现阶段**
   - 核心功能实现
   - 交互逻辑设置
   - 视觉效果配置

4. **验证与清理阶段**
   - 结果验证
   - 资源清理
   - 错误处理

## 示例转换

### 示例1：模糊需求转换

**原始需求**：
"我想做一个可以显示3D建筑，然后点击建筑物能高亮显示的功能。"

**结构化输出**：
```
# 需求明确化报告

## 原始需求
我想做一个可以显示3D建筑，然后点击建筑物能高亮显示的功能。

## 技术域分类
- 主要域: wdp-api-entity-general-behavior（实体操作）
- 次要域: wdp-api-initialization-unified（初始化）

## 关键参数
- 必要参数:
  * 服务器URL: 需要追问
  * 验证口令: 需要追问
  * 渲染容器ID: "player"（默认值）
  * 建筑物实体ID/类型: 需要追问
- 可选参数:
  * 高亮颜色: [1, 0, 0, 0.5]（默认红色半透明）
  * 高亮持续时间: 无限（直到取消高亮）

## 依赖关系
- 技术依赖: WdpApi, CloudApi
- 时序依赖: 初始化 → 场景就绪 → 注册点击事件 → 实体高亮
- 数据依赖: 点击事件 → 获取实体ID → 设置高亮

## 执行计划
1. 初始化阶段:
   - 加载SDK（CloudApi.min.js, wdpApi.min.js）
   - 创建WdpApi实例
   - 启动渲染器
   - 注册场景就绪事件
2. 准备阶段:
   - 等待场景完全加载（progress === 100）
   - 注册实体点击事件
3. 业务实现阶段:
   - 实现点击事件回调函数
   - 在回调中获取点击的实体
   - 设置实体高亮
   - 提供取消高亮的方法
4. 验证与清理阶段:
   - 验证高亮效果
   - 提供错误处理机制

## 缺失信息
1. 服务器URL是什么？
2. 验证口令是什么？
3. 具体是哪种类型的建筑物实体？
4. 是否需要特定的高亮颜色？
5. 点击后高亮是永久的还是临时的？
```

### 示例2：复杂需求转换

**原始需求**：
"做一个地图，上面显示一些标记点，点击标记可以弹出信息，还要能切换地图的2D和3D视图。"

**结构化输出**：
```
# 需求明确化报告

## 原始需求
做一个地图，上面显示一些标记点，点击标记可以弹出信息，还要能切换地图的2D和3D视图。

## 技术域分类
- 主要域: gis-api-core-operations（GIS操作）
- 次要域: 
  * wdp-api-coverings-unified（覆盖物/标记）
  * wdp-api-camera-unified（相机/视角切换）
  * wdp-api-initialization-unified（初始化）

## 关键参数
- 必要参数:
  * 服务器URL: 需要追问
  * 验证口令: 需要追问
  * 渲染容器ID: "player"（默认值）
  * 标记点数据: 需要追问（坐标和信息内容）
- 可选参数:
  * 标记样式: 默认样式
  * 信息窗口样式: 默认样式
  * 2D/3D切换动画时长: 1秒（默认值）

## 依赖关系
- 技术依赖: WdpApi, CloudApi, GisApi
- 时序依赖: 
  * 初始化 → 场景就绪 → GIS插件安装 → 地图加载 → 添加标记 → 注册事件
  * 2D/3D切换: 用户触发 → 相机视角变换
- 数据依赖: 
  * 标记数据 → 标记创建
  * 点击事件 → 获取标记ID → 显示信息窗口

## 执行计划
1. 初始化阶段:
   - 加载SDK（CloudApi.min.js, wdpApi.min.js, gisApi.min.js）
   - 创建WdpApi实例
   - 启动渲染器
   - 注册场景就绪事件
2. 准备阶段:
   - 等待场景完全加载（progress === 100）
   - 安装GIS插件
   - 加载地图底图
3. 业务实现阶段:
   - 创建标记点
   - 注册标记点击事件
   - 实现信息窗口显示逻辑
   - 实现2D/3D视图切换功能
4. 验证与清理阶段:
   - 验证标记点显示
   - 验证点击交互
   - 验证视图切换
   - 提供错误处理机制

## 缺失信息
1. 服务器URL是什么？
2. 验证口令是什么？
3. 标记点的数据来源是什么？包含哪些坐标和信息？
4. 地图初始中心点应该设置在哪里？
5. 信息窗口需要显示哪些具体内容？
6. 2D和3D视图切换是否需要特定的视角参数？
```

## 质量门槛

1. 必须识别出所有必要参数，并明确标记缺失的参数。
2. 必须考虑完整的依赖链，特别是初始化和场景就绪的前置依赖。
3. 执行计划必须包含验证和错误处理步骤。
4. 对于复杂需求，必须拆分为清晰的子任务。
5. 对每个可切换状态，确认存在开启路径 + 关闭路径 + toggle 支持 + 冲突处理。
6. 删除/卸载等不可逆操作必须有确认步骤。

## 高频问题

1. **参数不足**
   - 问题：用户提供的信息不足以实现需求
   - 解决：使用ask_followup_question工具提问，获取必要信息

2. **需求冲突**
   - 问题：用户需求中存在相互冲突的部分
   - 解决：明确指出冲突点，提供可行的替代方案

3. **技术限制**
   - 问题：用户需求超出WDP API能力范围
   - 解决：说明技术限制，提供最接近的可行方案

4. **性能考量**
   - 问题：用户需求可能导致性能问题
   - 解决：提出优化建议，如批量处理、减少实时计算等

## 输出要求

2. 缺失信息列表（如有）
3. 可能的技术限制或性能考量（如有）
4. 推荐的实现路径

## 强制性要求（再次强调）

1. 必须先明确所有必要参数，再进行代码生成
2. 必须考虑完整的依赖链和执行顺序
3. 必须包含验证和错误处理机制
4. 对于复杂需求，必须拆分为清晰的子任务

需求明确化是代码生成的前置步骤，只有将模糊需求转化为结构化的技术规格，才能生成高质量、可靠的代码实现。
